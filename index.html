00 — In your Slack workspace

Where cc-bot earns its keep.

Five everyday channels, five jobs the bot quietly handles. Nothing theatrical — just the loop of "ask in chat, get a real answer, keep working."

  1. #bugs

    PM drops a screenshot, cc-bot opens a fix PR

    Maya pastes a screenshot of a 500 from staging into #bugs and writes "@cc-bot what's going on?"

    cc-bot downloads the image, greps the error in the repo, traces it to a stale migration, opens a PR with the fix, and replies in the same thread with the diff and a link.

    Why it wins: the conversation, the screenshot, the fix, and the PR all live in one thread anyone can scroll back to.

  2. #deploys

    Solo dev ships from the channel

    You finish a feature on the train and type "ship to prod" into #deploys.

    Because deploy_prod is tagged admin-confirm, cc-bot replays the command and waits for "yes". It runs the deploy, streams the build URL, and posts the version tag when it's live.

    Why it wins: no SSH, no dashboard tab, no terminal — and still a full audit trail in Slack.

  3. #product

    Cross-team status without pinging the team

    In #product a PM asks "where are we on Feature X?"

    cc-bot reads the project's progress doc (path lives in the profile), summarizes the open tasks and last commits touching the feature flag, and posts a tidy bullet list.

    Why it wins: the engineer keeps their flow, the PM gets their answer in 20 seconds.

  4. #standup

    Async standup that writes itself

    At 09:30 a scheduled prompt drops in #standup: "what shipped yesterday?"

    cc-bot diffs the last 24h of git log, cross-refs the intents in profile.intents, and posts a three-line summary per repo with author handles linked.

    Why it wins: the standup happens whether or not the humans show up.

  5. #ops-alerts

    On-call gets a triage, not just an alert

    Datadog fires a latency alert into #ops-alerts at 02:14.

    cc-bot picks up the keyword, queries logs through the project's configured MCP, and threads a probable cause: "p95 spike correlates with the new orders index migration — suggest pausing reindex job."

    Why it wins: the on-call wakes up to a starting point, not a mystery.

01 — Capabilities

Six muscles, one bot.

  1. 01

    Slack-native, IM-agnostic

    Slack ships with Socket Mode push (no public webhook needed, works behind firewalls). The same IMAdapter interface also powers the Lark adapter — and the next adapter is one file.

    • Slack
    • Lark
    • Socket Mode
  2. 02

    Defensive delivery

    Slack push events get triangulated against a short-poll fallback and a deduplicated emit ledger, so every channel message is delivered exactly once even when the WebSocket hiccups.

    • push
    • poll fallback
    • dedupe ledger
  3. 03

    Multi-agent scheduling

    Slots and tags turn parallel Claude sessions into a polite queue. Two deploys won't fight each other; a query and a deploy can run side by side. Conflicts resolved by config, not chance.

    • slot
    • tag
    • conflict matrix
  4. 04

    HUD status broadcast

    The Claude Code statusline becomes a live indicator: which agents are busy, which slots are free, last channel thread. Your terminal shows the same truth the Slack channel shows.

    • statusline
    • real-time
    • shared state
  5. 05

    Three-tier permission matrix

    Public reads, admin-auto writes, admin-confirm for the scary ones. Each intent is tagged once in the profile; the bot enforces it forever. The channel stays open, the repo stays safe.

    • public
    • admin-auto
    • admin-confirm
  6. 06

    Main-window priority

    A UserPromptSubmit hook lets the developer at the keyboard preempt any background bot task. When you're focused, the bot waits — when you step away, it gets back to work.

    • hook
    • preempt
    • focus-aware

02 — Engineering note

One IMAdapter, many group chats.

adapters/base.js defines the contract — listRecentMessages, sendText, sendImage, downloadResource, getUser. Each IM is a single file that fulfils that contract. Slack and Lark ship today; Discord, Teams, or your private chat platform is a weekend project.

The runtime reads profile.im.type at startup, instantiates the right adapter, and the rest of the pipeline never knows which IM it's talking to. Channels are channels, messages are messages, replies go back the way they came. 

// adapters/base.js — the contract every IM implements
class IMAdapter {
  listRecentMessages(chat_id, since_ts) { }
  sendText(chat_id, text)             { }
  sendImage(chat_id, file_path)       { }
  downloadResource(file_id, out_path) { }
  getUser(user_id)                    { }
}

// runtime/poll.js — IM is a config field
const adapter = loadAdapter(profile.im.type)
// → 'slack' loads adapters/slack.js
// → 'lark'  loads adapters/lark.js

03 — Architecture

A small idea, drawn carefully.

The bot is a thin pipeline. Slack channel events arrive on Socket Mode (Lark uses HTTP polling). A short-poll fallback and a dedupe ledger guarantee exactly-once. The scheduler hands messages to a Claude Code session under the right permission tier, and the result flows back out — to the channel and to the HUD at the same time.

04 — Slack setup

Four steps to a chat-driven repo.

The /cc-bot:setup slash command runs the whole flow inside Claude Code — it's idempotent, version-aware, and auto-detects whatever it can. Below is what it walks you through.

  1. 01

    Install the cc-bot plugin

    # inside Claude Code
/plugin marketplace add WaterTian/cc-bot
/plugin install cc-bot@WaterTian-cc-bot

    Two lines, then run /cc-bot:setup in any project directory. When it asks "which IM?", pick Slack.

  2. 02

    Create the Slack app from the manifest

    cc-bot ships templates/slack-manifest.yaml. Open api.slack.com/apps → Create New App → From a manifest, pick your workspace, paste the YAML below, then Create. 

    # templates/slack-manifest.yaml — bot scopes & Socket Mode
display_information:
  name: cc-bot

features:
  bot_user:
    display_name: cc-bot
    always_online: true

oauth_config:
  scopes:
    bot:
      - app_mentions:read   # listen for @cc-bot
      - channels:history    # read public channels
      - channels:read
      - chat:write          # post messages
      - files:read          # download images
      - files:write
      - groups:history      # private channels
      - groups:read
      - users:read

settings:
  event_subscriptions:
    bot_events:
      - message.channels
      - message.groups
      - app_mention
  socket_mode_enabled: true  # no public webhook needed

  3. 03

    Generate & paste the two tokens

    In the new app's Basic Information page:

    • App-Level Token — generate one named cc-bot-socket with scope connections:write. Copy the xapp-1-… value.
    • Install App → Install to Workspace → Allow. Copy the Bot User OAuth Token (xoxb-…).

    Paste both tokens when /cc-bot:setup prompts you. cc-bot calls auth.test to verify, and writes them into .cc-bot/profiles/active.json (already gitignored).

  4. 04

    Invite cc-bot to a channel

    Paste the channel ID (e.g. C0XXXXXXXXX) when asked. In Slack, run /invite @cc-bot in that channel — cc-bot sends a probe message to confirm. Then bring it online: 

    /cc-bot:start

    You'll see an "online" message land in your channel. From here on, talk to it like you'd talk to any teammate.

Slash commands

  • /cc-bot:setup — interactive Slack / Lark onboarding
  • /cc-bot:start — bring the bot online (Socket Mode connects)
  • /cc-bot:stop — take it offline
  • /cc-bot:new-profile <name> — clone the profile template
  • /cc-bot:switch <name> — swap active profile
  • /cc-bot:doctor — health check & diagnostics
